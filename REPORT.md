## Раздел 1: Описание проблемы

### Что такое Race Condition?

_TODO: Объясните, что такое race condition в контексте баз данных._

Race condition (состояние гонки) — это ситуация, когда итоговое состояние данных зависит от порядка и времени выполнения нескольких параллельных транзакций. Если две или более транзакции одновременно читают и изменяют одни и те же данные без достаточной синхронизации, система может прийти к некорректному результату.

**Пример из жизни:**
- Два покупателя одновременно пытаются купить последний товар на складе.
- Оба приложения видят, что остаток товара равен 1.
- Оба оформляют заказ и уменьшают остаток.
- В итоге система подтверждает два заказа на один товар.

### Почему READ COMMITTED не защищает от двойной оплаты?

READ COMMITTED гарантирует только то, что каждая операция чтения внутри транзакции видит только уже зафиксированные (COMMIT) данные. Но он не блокирует строку при обычном SELECT. Поэтому две транзакции могут почти одновременно прочитать одно и то же состояние заказа (status = 'created') и обе решить, что заказ ещё можно оплатить.

На уровне изоляции READ COMMITTED:
1. Каждая транзакция видит только зафиксированные данные на момент выполнения конкретного SQL-запроса.
2. Когда две транзакции одновременно читают один и тот же заказ до его обновления, обе получают одинаковое значение status = 'created'.
3. В результате обе транзакции считают оплату допустимой и пытаются изменить статус заказа на paid.

**Демонстрация проблемы:**

```
Время | Сессия 1                    | Сессия 2
------|----------------------------|---------------------------
t1    | BEGIN                      |
t2    | SELECT status (created)    |
t3    |                            | BEGIN
t4    |                            | SELECT status (created)
t5    | pg_sleep(3)                |
t6    |                            | pg_sleep(3)
t7    | UPDATE status = paid       |
t8    |                            | UPDATE status = paid (ждет)
t9    | COMMIT                     |
t10   |                            | UPDATE выполняется...
t11   |                            | COMMIT
```

Пояснение по этапам:
- t1 — первая транзакция начинается.
- t2 — Сессия 1 читает статус заказа и видит created.
- t3 — вторая транзакция начинается.
- t4 — Сессия 2 читает статус заказа и тоже видит created, потому что Сессия 1 ещё не закоммитила изменения.
- t5/t6 — транзакции искусственно задерживаются.
- t7 — Сессия 1 выполняет UPDATE, меняет статус заказа на paid и получает блокировку на строку.
- t8 — Сессия 2 тоже пытается выполнить UPDATE, но ждёт освобождения блокировки строки.
- t9 — Сессия 1 делает COMMIT, изменение становится видимым другим транзакциям.
- t10 — Сессия 2 продолжает выполнение UPDATE. Логика проверки была сделана до UPDATE, приложение уже «решило», что заказ можно оплатить.
- t11 — Сессия 2 завершает транзакцию, и в системе появиться повторная запись в историю.

### Примеры из реальной жизни

_TODO: Приведите реальные примеры, когда может возникнуть такая проблема._

1. **Двойной клик на кнопку "Оплатить"**
   - Пользователь нажимает кнопку дважды
   - Два HTTP-запроса приходят почти одновременно
   - Оба запроса начинают обработку параллельно

2. **Микросервисная архитектура**
   - Один сервис инициирует оплату через API
   - Второй сервис параллельно обрабатывает callback от платёжной системы
   - Оба сервиса пытаются изменить статус одного и того же заказа
   - Без блокировок возможно повторное проведение операции

3. **Сбой сети**
   - Клиент отправляет запрос на оплату
   - Ответ от сервера теряется из-за сетевого сбоя
   - Клиент считает, что операция не выполнена, и повторяет запрос
   - Сервер получает два почти одинаковых запроса и может обработать оба

## Раздел 2: Уровни изоляции в PostgreSQL

### READ UNCOMMITTED

**Описание:**
На уровне READ UNCOMMITTED транзакция теоретически может читать данные, которые были изменены другой транзакцией, но ещё не зафиксированы (dirty read).

**Предотвращает:**
- Формально по стандарту SQL практически ничего не гарантирует

**Не предотвращает:**
- Dirty reads
- Non-repeatable reads
- Phantom reads

**Когда использовать:**
В PostgreSQL отдельно использовать его смысла нет. В других СУБД его могут применять для максимально «дешёвого» чтения, когда допустимы неточные и временно неконсистентные данные.

**Особенность в PostgreSQL:**
В PostgreSQL READ UNCOMMITTED работает как READ COMMITTED из-за архитектуры MVCC.

---

### READ COMMITTED (по умолчанию)

**Описание:**
_TODO: Опишите этот уровень изоляции._

Каждый SELECT в транзакции видит только те данные, которые были зафиксированы к моменту начала именно этого запроса.

**Предотвращает:**
- ✅ Dirty reads (чтение незакоммиченных данных)

**Не предотвращает:**
- ❌ Non-repeatable reads (повторное чтение дает другой результат)
- ❌ Phantom reads (новые строки появляются в результате запроса)

**Пример non-repeatable read:**
```sql
-- Сессия 1
BEGIN;
SELECT balance FROM accounts WHERE id = 1; -- Результат: 1000
-- Сессия 2 изменяет balance и делает COMMIT
SELECT balance FROM accounts WHERE id = 1; -- Результат: 500 (!)
COMMIT;
```

**Когда использовать:**
Подходит для большинства повседневных операций:

- чтение каталога товаров;
- просмотр профиля пользователя;
- отображение истории заказов;
- обновление некритичных данных;
- операции, где допустимо, что между двумя SELECT внутри одной транзакции данные могут измениться.

---

### REPEATABLE READ

**Описание:**

Транзакция видит snapshot данных на момент начала начала транзакции.

**Предотвращает:**
- ✅ Dirty reads
- ✅ Non-repeatable reads
- ✅ Phantom reads (в PostgreSQL благодаря MVCC)

**Не предотвращает:**
- ❌ Write skew (специфичная аномалия)
- ❌ Некоторые сериализационные аномалии

**Особенность в PostgreSQL:**
В отличие от стандарта SQL, PostgreSQL на REPEATABLE READ также предотвращает phantom reads.

**Когда использовать:**

Подходит для операций, где важно, чтобы логика выполнялась на стабильном наборе данных:

- обработка оплаты;
- резервирование товара;
- расчёт баланса;
- генерация финансовых отчётов в рамках одной транзакции;
- операции, где несколько проверок и обновлений должны опираться на одну и ту же версию данных.

---

### SERIALIZABLE

**Описание:**

Самый строгий уровень, гарантирующий такое поведение, как будто все транзакции выполнялись не параллельно, а строго последовательно одна за другой.

**Предотвращает:**
- ✅ Все аномалии чтения
- ✅ Сериализационные аномалии
- ✅ Write skew

**Недостатки:**
- ❌ Может откатывать транзакции при конфликтах (serialization failure)
- ❌ Снижение производительности
- ❌ Требует retry logic в приложении

**Пример serialization failure:**
```sql
-- Сессия 1
BEGIN ISOLATION LEVEL SERIALIZABLE;
SELECT SUM(balance) FROM accounts; -- 5000
UPDATE accounts SET balance = balance + 100 WHERE id = 1;

-- Сессия 2 (параллельно)
BEGIN ISOLATION LEVEL SERIALIZABLE;
SELECT SUM(balance) FROM accounts; -- 5000
UPDATE accounts SET balance = balance + 200 WHERE id = 2;

-- При COMMIT одна из транзакций получит ошибку:
-- ERROR: could not serialize access due to read/write dependencies
```

**Когда использовать:**
Подходит для операций, где важно, чтобы логика выполнялась на стабильном наборе данных:

- банковские и финансовые транзакции;
- расчёт лимитов и квот;
- взаимосвязанные изменения нескольких сущностей;
- бизнес-правила, где даже редкие аномалии недопустимы.

_*В PostgreSQL REPEATABLE READ также предотвращает phantom reads благодаря MVCC._

## Раздел 3: Решение проблемы

### Почему REPEATABLE READ решает проблему?

REPEATABLE READ использует snapshot isolation:
1. Транзакция видит одну и ту же версию данных на всём протяжении выполнения.
2. Это означает, что повторные чтения не «прыгают» между разными состояниями БД, и бизнес-логика работает предсказуемо.
3. Однако, без блокировок две транзакции всё равно могут одновременно прочитать одну и ту же строку и попытаться изменить её независимо, если не использовать механизмы синхронизации.

### Зачем нужен FOR UPDATE?

FOR UPDATE нужен для того, чтобы не просто прочитать строку, а сразу заблокировать её для конкурентных изменений. 
Это превращает «проверку + изменение» в защищённую критическую секцию.

`FOR UPDATE` создает эксклюзивную блокировку на уровне строки (row-level lock):

**Без FOR UPDATE:**
```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...';
-- Другая транзакция может прочитать ту же строку
UPDATE orders SET status = 'paid' WHERE id = '...';
COMMIT;
```

**С FOR UPDATE:**
```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...' FOR UPDATE;
-- Другая транзакция ЖДЕТ освобождения блокировки
UPDATE orders SET status = 'paid' WHERE id = '...';
COMMIT;
```

**Типы блокировок:**
- `FOR UPDATE` — эксклюзивная блокировка (никто не может изменить или заблокировать)
- `FOR SHARE` — разделяемая блокировка (другие могут читать с FOR SHARE, но не могут изменять)
- `FOR NO KEY UPDATE` — как FOR UPDATE, но разрешает concurrent FOR KEY SHARE
- `FOR KEY SHARE` — слабая блокировка (предотвращает DELETE и UPDATE ключевых полей)

### Что произойдет без FOR UPDATE на REPEATABLE READ?

Если использовать REPEATABLE READ, но не блокировать строку через FOR UPDATE, обе транзакции могут успешно прочитать одно и то же исходное состояние (status = 'created') и попытаться провести оплату независимо друг от друга.

Даже на REPEATABLE READ возможна аномалия:

```sql
-- Сессия 1
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...'; -- created
-- Задержка...
UPDATE orders SET status = 'paid' WHERE id = '...' AND status = 'created';
COMMIT;

-- Сессия 2 (параллельно)
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...'; -- created (snapshot!)
-- Задержка...
UPDATE orders SET status = 'paid' WHERE id = '...' AND status = 'created';
-- ЧТО ПРОИЗОЙДЕТ?
```
Обе транзакции прочитают status = 'created', потому что у каждой свой snapshot на момент BEGIN.
Сессия 1 первой выполнит UPDATE и закоммитит изменение.
Сессия 2 при попытке обновления столкнётся с конфликтом версии строки.
В PostgreSQL на REPEATABLE READ вторая транзакция, как правило, получит ошибку вида:
```
ERROR: could not serialize access due to concurrent update
```

### Разница между FOR UPDATE и FOR SHARE

| Характеристика | FOR UPDATE | FOR SHARE |
|----------------|------------|-----------|
| Тип блокировки | Эксклюзивная | Разделяемая |
| Блокирует чтение с FOR UPDATE | ✅ | ✅ |
| Блокирует чтение с FOR SHARE | ✅ | ❌ |
| Блокирует UPDATE | ✅ | ✅ |
| Блокирует DELETE | ✅ | ✅ |
| Use case | Перед изменением | Защита от изменений |

**Пример использования FOR SHARE:**
```sql
-- Проверка наличия товара перед созданием заказа
SELECT quantity FROM products WHERE id = '...' FOR SHARE;
-- Товар не может быть удален или изменен до COMMIT
```

## Раздел 4: Рекомендации для продакшена

### Какой ISOLATION LEVEL использовать для продакшена маркетплейса?

**Рекомендация:** Для продакшена маркетплейса рекомендуется использовать **READ COMMITTED как default уровень** с явным использованием **REPEATABLE READ + FOR UPDATE для критичных операций**.

#### Обоснование:

**1. Производительность**
- READ COMMITTED имеет минимальный overhead
- REPEATABLE READ используется только там, где необходимо
- Избегаем глобального использования SERIALIZABLE

Большинство запросов в маркетплейсе — это чтение: каталог, карточки товаров, история заказов, фильтрация, поиск.
Для таких сценариев нет смысла повышать уровень изоляции глобально.
Чем меньше долгоживущих транзакций и тяжёлых блокировок, тем выше пропускная способность БД и ниже latency.

**2. Безопасность данных**
- Критичные операции (оплата, изменение баланса) защищены через REPEATABLE READ + FOR UPDATE
- Некритичные операции (просмотр каталога) работают на READ COMMITTED
- Четкое разделение зон ответственности

Такой подход позволяет защищать именно те участки системы, где ошибка действительно дорого стоит.
Например, повторная оплата, двойное списание средств, продажа последнего товара двум клиентам — это уже бизнес-критичные инциденты.
Важно не «усложнять всё», а делать безопасность адресной и осознанной.

**3. Риски deadlock**
- FOR UPDATE может привести к deadlock при неправильном порядке блокировок
- Решение: всегда блокировать ресурсы в одном порядке
- Пример: сначала order, потом order_items, потом payment

Deadlock — это не «ошибка PostgreSQL», а следствие конкурирующей логики в приложении.
Его можно существенно снизить дисциплиной проектирования:
всегда брать блокировки в одном порядке,
держать транзакции короткими,
не выполнять внешние HTTP/API-вызовы внутри открытой транзакции.

**4. Простота разработки**
- READ COMMITTED легко понять и отлаживать
- REPEATABLE READ + FOR UPDATE требует explicit intent
- Код становится самодокументируемым

Когда разработчик видит SELECT ... FOR UPDATE, сразу понятно, что это критичная секция.
Команда быстрее понимает, где возможна конкуренция, а где она несущественна.

**5. Масштабируемость**
- READ COMMITTED хорошо масштабируется
- Локальное использование блокировок не создает bottleneck
- Connection pooling работает эффективно

Если блокировки применяются точечно, они не становятся глобальным ограничением для всей системы.
Это особенно важно для маркетплейса, где одновременно работают пользователи, продавцы, платёжные сервисы, фоновые задачи и интеграции.

### Альтернативные подходы

#### Подход 1: Использовать SERIALIZABLE везде

**Плюсы:**
- Максимальная корректность
- Не нужно думать о блокировках
- PostgreSQL автоматически обнаруживает конфликты

**Минусы:**
- Значительное снижение производительности (20-50%)
- Требует retry logic во всех операциях
- Высокий процент rollback при нагрузке
- Сложная отладка serialization failures

**Вывод:** Не рекомендуется для high-load систем.

#### Подход 2: Optimistic Locking (версионирование)

**Реализация:**
```sql
ALTER TABLE orders ADD COLUMN version INTEGER DEFAULT 1;

UPDATE orders 
SET status = 'paid', version = version + 1
WHERE id = '...' AND status = 'created' AND version = 1;

-- Проверить ROW_COUNT, если 0 — конфликт
```

**Плюсы:**
- Нет блокировок на чтение
- Хорошая производительность
- Масштабируется горизонтально

**Минусы:**
- Требует изменения схемы БД
- Требует retry logic в приложении
- Может быть много конфликтов при высокой нагрузке

**Вывод:** Хороший подход для распределенных систем.

#### Подход 3: Advisory Locks

**Реализация:**
```sql
BEGIN;
SELECT pg_advisory_xact_lock(hashtext('order_' || order_id));
-- Критическая секция
COMMIT; -- Блокировка автоматически снимается
```

**Плюсы:**
- Гибкий контроль блокировок
- Можно блокировать логические ресурсы
- Работает на любом уровне изоляции

**Минусы:**
- Легко забыть снять блокировку
- Сложнее отлаживать
- Требует дисциплины от разработчиков

**Вывод:** Полезно для специфичных случаев, но не как основной подход.

### Итоговая рекомендация

Для продакшена маркетплейса используйте **гибридный подход**:

1. **Default isolation level:** `READ COMMITTED`
   - Для обычных операций (чтение каталога, просмотр истории)
   - 95% операций

2. **REPEATABLE READ + FOR UPDATE:** для критичных операций
   - Оплата заказа
   - Изменение баланса
   - Резервирование товара
   - 5% операций

3. **Дополнительно:** Optimistic locking для операций с высокой конкурентностью
   - Обновление счетчиков
   - Изменение рейтингов

**Пример кода:**
```python
async def pay_order(order_id: UUID):
    async with db.transaction(isolation='repeatable_read'):
        # Заблокировать заказ
        order = await db.fetch_one(
            "SELECT * FROM orders WHERE id = $1 FOR UPDATE",
            order_id
        )
        
        if order['status'] != 'created':
            raise OrderAlreadyPaidError()
        
        # Обновить статус
        await db.execute(
            "UPDATE orders SET status = 'paid' WHERE id = $1",
            order_id
        )
        
        # Записать в историю
        await db.execute(
            "INSERT INTO order_status_history (order_id, status) VALUES ($1, 'paid')",
            order_id
        )
```
